Data Hierarchy

There are four levels at which session data is aggregated. Higher levels are derived by rolling up lower levels.

Chapter Full window of sessions since the last checkpoint. All days combined.
Day All sessions on a given calendar date — includes path workouts + explore (freeplay).
Workout 1–3 classes from the user's structured daily path. Requires workout identifier field
Class Single session. One row in the session CSV. All raw fields available.

Data Sources

All per-session fields come from the raw session CSV. Cross-chapter comparisons use the equivalent window from the previous chapter.

Derived Aggregates

Higher-level metrics are computed by rolling up raw session fields. Class-level is always raw — no derivation needed.

Workout level

Group sessions by workout ID (TBD). Sum or max across the 1–3 classes in that workout.

workout.calories = sum(calories) across classes in workout
workout.minutes = sum(time) / 60 across classes in workout
workout.movePoints = sum(points) across classes in workout
workout.totalPunches = sum(all punch fields) across classes in workout

Day level

Group sessions by calendar date from dateCreation. Sum across all classes that day.

day.calories = sum(calories) — all sessions on that date
day.minutes = sum(time) / 60 — all sessions on that date
day.movePoints = sum(points) — all sessions on that date
day.sessionCount = count of sessions on that date

Chapter level

Aggregate across the full chapter window (all sessions since last checkpoint).

classesCompleted = count of session rows in chapter
workoutsCompleted = count of distinct workouts in chapter
totalMinutes = sum(time) / 60
caloriesBurned = sum(calories)
movePoints = sum(points)
sessionsByClassType = group by classType, sort by count desc
streakDays = binary array: 1 for each day with ≥1 session
streakWeeks = longest run of 7-day windows with ≥ N active days

Highlight Logic

The highlight is the single most impressive, concrete number from the chapter. It must feel like a specific memory, not a summary metric.

💡

What makes a good highlight: Concrete and human-readable (312 punches, not performance score 72.4). Specific to a session. Varies across checkpoints so it doesn't go stale.

Selection algorithm

1

Filter

Remove candidates where key fields are zero/empty across the chapter. Remove candidates whose granularity condition isn't met (e.g. no multi-class workouts → skip workout-level candidates, no multi-session days → skip day-level candidates).

2

Score

For each eligible candidate, compute a normalized score:

score = value / impressive_threshold
# Special case H16 (reaction time): score = threshold / value (lower is better)
3

Pick winner

Highest score wins. On ties, prefer more specific granularity: Class > Workout > Day > Chapter.

4

Variety check

If the same category won last checkpoint, skip it and use the next-highest scorer instead. Tracks which category was shown previously.

5

Format sub-label

Class-level → "{displayClassType}, {formattedDate}"
Workout-level → "Workout, {formattedDate}"
Day-level → "{formattedDate}"
Chapter-level → "Across {N} classes"

Effort — Time, MP, Calories
Wrapped Moment
Design ref

Aggregate effort for the chapter with chapter-over-chapter deltas. All three stats are chapter-level totals.

Element Variable Logic
Workout time totalMinutes sum(time) / 60 rounded
Time delta totalMinutes - previousTotalMinutes Shown if previousTotalMinutes > 0
Move Points movePoints sum(points) — chapter total
Calories caloriesBurned sum(calories) — chapter total
Calorie delta caloriesBurned - previousCaloriesBurned Shown if previousCaloriesBurned > 0
Percentile badge percentile Server-side. Shown only if percentile ≤ 10
ℹ

Delta format: ↑ {N} mins / ↑ {N} kcal
Percentile copy: "Top {N}% of FitXR users this chapter"
Move Points delta: Currently no delta shown — revisit if it adds value.

Badge Logic

Mini-achievements surfaced across the Wrapped Moment. Each badge must feel earned and specific — not generic. The percentile badge in particular needs a universal, fixed reference window so comparisons are meaningful across users whose chapters are all personalised and time-shifted.

Badge types

B1
Community rank
Top {N}% — calendar window

Percentile ranking against all FitXR users on a fixed, universal time window. Avoids the personalised-chapter comparison problem.

Label: "Top {N}% in {Month}"
Window: calendar month (same for all users)
Metric: Move Points, calories, or classes — pick highest percentile
B2
Peak session rank
Your best session ranked in the top {N}%

Compares a single peak session against all users globally — sidesteps the chapter window problem entirely and feels more personal.

Label: "Your best session: top {N}% for Move Points globally"
Metric: max(points) in a single session this chapter
B3
Personal best
New PB on any tracked metric

User beat their own all-time record on a specific metric this chapter. Purely self-comparative — no community baseline needed.

Examples: "New PB — fastest punch ever", "Best combo this chapter"
Trigger: value > lifetime max on that metric
B4
Chapter-over-chapter growth
More active / stronger than last chapter

Celebrates meaningful improvement vs the previous chapter. Works within the personalised window since both chapters belong to the same user.

Examples: "20% more active than last chapter", "Doubled your Move Points"
Trigger: delta > threshold vs previousChapter
B5
Streak
{N}-day or {N}-week streak

Consistency milestone. Already surfaced on the Streak screen — surfacing as a badge here too gives it cross-screen weight.

Examples: "7-day streak", "First 30-day streak"
Trigger: streakDays crosses a milestone threshold
B6
Lifetime volume milestone
Cumulative threshold crossed

Celebrates crossing a round lifetime number. Works independently of any time window.

Examples: "500 classes completed", "10,000 punches landed", "1,000 mins worked out"
Trigger: lifetime cumulative crosses defined threshold
B7
Consistency pattern
Showed up every day this week / hit goal every week

Pattern-based rather than raw volume — rewards regularity over intensity.

Examples: "Worked out every day this week", "Hit your goal 4 weeks running"
Trigger: pattern condition met within chapter window
B8
Effort spike
Busiest chapter yet / most calories in a single day

Chapter-relative personal best — compares this chapter's peak against all previous chapters.

Examples: "Busiest chapter yet", "Most calories burned in a single day — ever"
Trigger: chapter aggregate exceeds previous chapter max

Intensity Suggestion
Plan Reset
Design ref

Presents the recommendation engine's output to the user in a motivating, human way. The recommendation itself is fully determined upstream — this screen just surfaces the result through pre-written copy variants and asks the user to confirm or opt out.

Copy variants

Three pre-written variants cover all possible recommendation engine outcomes — intensity increased, stays the same, or decreased. Full copy TBD.

User agency — CTAs

CTAs are only shown when the recommendation involves a change. If the plan stays the same, no confirmation is needed and the screen flows through automatically.

Recommendation CTAs shown
Intensity increased or decreased "Let's lock in!" to accept · "Keep it as it is" to opt out
Intensity stays the same Simple CTA to acknowledge the recommendation
ℹ

Giving the user an explicit opt-in for changes — rather than applying them automatically — reinforces their sense of control and makes them feel like an active participant in their plan.
