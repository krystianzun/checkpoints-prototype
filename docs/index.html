Data Hierarchy

There are four levels at which session data is aggregated. Higher levels are derived by rolling up lower levels.

Chapter Full window of sessions since the last checkpoint. All days combined.
Day All sessions on a given calendar date — includes path workouts + explore (freeplay).
Workout 1–3 classes from the user's structured daily path. Requires workout identifier field
Class Single session. One row in the session CSV. All raw fields available.

Data Sources

All per-session fields come from the raw session CSV. Cross-chapter comparisons use the equivalent window from the previous chapter.

Derived Aggregates

Higher-level metrics are computed by rolling up raw session fields. Class-level is always raw — no derivation needed.

Workout level

Group sessions by workout ID (TBD). Sum or max across the 1–3 classes in that workout.

workout.calories = sum(calories) across classes in workout
workout.minutes = sum(time) / 60 across classes in workout
workout.movePoints = sum(points) across classes in workout
workout.totalPunches = sum(all punch fields) across classes in workout

Day level

Group sessions by calendar date from dateCreation. Sum across all classes that day.

day.calories = sum(calories) — all sessions on that date
day.minutes = sum(time) / 60 — all sessions on that date
day.movePoints = sum(points) — all sessions on that date
day.sessionCount = count of sessions on that date

Chapter level

Aggregate across the full chapter window (all sessions since last checkpoint).

classesCompleted = count of session rows in chapter
workoutsCompleted = count of distinct workouts in chapter
totalMinutes = sum(time) / 60
caloriesBurned = sum(calories)
movePoints = sum(points)
sessionsByClassType = group by classType, sort by count desc
streakDays = binary array: 1 for each day with ≥1 session
streakWeeks = longest run of 7-day windows with ≥ N active days

Highlight Logic

The highlight is the single most impressive, concrete number from the chapter. It must feel like a specific memory, not a summary metric.

💡

What makes a good highlight: Concrete and human-readable (312 punches, not performance score 72.4). Specific to a session. Varies across checkpoints so it doesn't go stale.

Selection algorithm

1

Filter

Remove candidates where key fields are zero/empty across the chapter. Remove candidates whose granularity condition isn't met (e.g. no multi-class workouts → skip workout-level candidates, no multi-session days → skip day-level candidates).

2

Score

For each eligible candidate, compute a normalized score:

score = value / impressive_threshold
# Special case H16 (reaction time): score = threshold / value (lower is better)
3

Pick winner

Highest score wins. On ties, prefer more specific granularity: Class > Workout > Day > Chapter.

4

Variety check

If the same category won last checkpoint, skip it and use the next-highest scorer instead. Tracks which category was shown previously.

5

Format sub-label

Class-level → "{displayClassType}, {formattedDate}"
Workout-level → "Workout, {formattedDate}"
Day-level → "{formattedDate}"
Chapter-level → "Across {N} classes"

Class type display names

Combat→Combat
Hiit→HIIT
Aero→Aero
Sculpt→Sculpt
Dance→Dance

Intensity Suggestion
Plan Reset

Adaptive screen that tells the user what the system recommends for the next chapter. Two variants based on performance.

Variant A — "You're KILLING it recently!"
  • classesCompleted ≥ 8 in this chapter
  • streakWeeks ≥ 2
  • Chapter-over-chapter calorie delta > 0

Variant A copy: "Let's increase your intensity. You can handle a little extra spice in your next set of workouts. We're going to push the intensity a little more — you've got this!"

Variant B — Standard (everyone else)
  • Standard transition. No intensity recommendation.
⚠

Open Q: What does "increase intensity" concretely change? Workout length tier? Difficulty flag? Class type recommendation? See Open Questions #12.

Open Questions

Unresolved questions that must be answered before the logic can be fully implemented. Blockers are flagged.

#1

Workout identifier

No workoutId field in the current CSV. Options: (a) add workoutId to session data, (b) group by date + consecutive timestamp proximity, (c) treat all path classes on the same day as one workout. Blocks H3, H6, H10.

#2

Path vs explore sessions

Is there a field that distinguishes path/journey classes from explore (freeplay) classes? Needed to attribute sessions to workouts correctly. classId references a studio class but doesn't indicate source.

#3

Chapter window definition

What defines a chapter? Fixed 2-week window? Rolling? Event-triggered by the user hitting a goal?

#4

dateCreation timezone

Unix timestamp — which timezone to display dates in? User's local, or UTC?

#5

percentile source

Pre-computed server-side, or derived from leagueRank / leagueTotalPlayers?

#6

medal field scale

Values seen in data: 1 and 4. Is it 1=none, 2=bronze, 3=silver, 4=gold? Direction should be confirmed.

#7

handTrackingPercent = 0

Does 0 mean perfect tracking (zero drops) or 0% tracked? Direction is unclear.

#8

Combo field semantics

Are bestExplosiveStreak, bestLegacyStreak, bestLightningStreak different names for the same combo mechanic in different class modes, or genuinely different move types? Affects whether we always take the max.

#9

Almosts — positive or negative signal?

Does a high almostsPerformed count mean the user pushed hard (positive) or struggled (neutral/negative)? Affects whether H20 is a valid highlight candidate.

#10

Streak definition

What counts as "one week streak"? 7 consecutive days? Or ≥ N sessions in any 7-day window?

#11

Achievements as game system

Are the milestone badges a pre-existing FitXR achievement system we surface, or are we defining them from scratch? Does the user collect them persistently?

#12

Highlight rotation persistence

How do we track which highlight category was shown last checkpoint to enforce variety? Stored server-side on the user profile?

#13

Highlight thresholds calibration

All thresholds in this doc are rough estimates. Need real distribution data to calibrate what's genuinely impressive vs ordinary.

#14

Intensity → plan mapping

What does "increase intensity" in Variant A concretely change? Workout length only, or a difficulty parameter in the class recommendation?
